# V3 #19 — UI / Tauri (agent console + Hub dashboard)

Shared React frontend (two role-views) + a locked-down Tauri v2 shell (design
§3.1, §7.1, §10 #5). **Implementation + frontend build are verified here; the
visual/UX pass and the native `cargo tauri build` run on the operator's machine.**

## What's in this issue
- `taskpaw_v3/ui/` — Vite + React 19 + MUI 6 frontend:
  - `theme.ts` from `design-system/taskpaw-v3/MASTER.md` (dark OLED, navy slate
    + blue accent, Fira Code/Sans) + status colors from the agent-console page.
  - `AgentConsole` (per `pages/agent-console.md`): two-pane — left rail of this
    machine's monitors with status dots, main pane = selected monitor's live
    status + metrics. Polls the agent `/status`.
  - `HubDashboard` (per `pages/hub-dashboard.md`): fleet grid of machines + the
    Hub's own host-health self-monitor. Polls the Hub `/status`.
  - `SchemaForm` (RJSF + MUI): schema-driven monitor config (design §4.3), fed by
    each plugin's `json_schema`.
  - `api.ts`: REST client; base URL + Bearer injected on the loopback origin in
    the shell, or via Vite env in the browser.
- `taskpaw_v3/src-tauri/` — Tauri v2 shell:
  - `tauri.conf.json`: `withGlobalTauri=false`, CSP restricted to the loopback
    backend + the Google-Fonts origin; bundles `../ui/dist`.
  - `capabilities/default.json`: **empty permissions** — no IPC/FS/shell exposed
    to the webview (§3.1). The UI is pure web → local backend over HTTP.
  - `src/main.rs`: spawns the backend child and ensures the **whole tree** dies on
    exit — Unix SIGTERMs the backend (its GracefulShutdown stops the supervisor +
    managed children) then force-kills if it lingers; Windows assigns the child to
    a **Job Object (KILL_ON_JOB_CLOSE)** so the OS terminates the tree when the
    shell exits. "X = exit", no orphan/held port (§7.1) — the V2 tray-zombie fix.
  - The window is created in `main.rs` (not config) so a runtime **init script**
    injects `window.__TASKPAW__ = {baseUrl, apiKey, role}` from the shell's env
    (`TASKPAW_UI_BASE` / `TASKPAW_UI_TOKEN` / `TASKPAW_UI_ROLE`) on the loopback
    origin — packaged builds don't rely on compile-time Vite env for the token.

> ⚠️ The Rust shell is **not compile-verified in this environment** (no cargo
> here). The operator must `cargo tauri build` on macOS AND Windows and verify:
> the app launches, X-close terminates the backend tree (no leftover process /
> port), and a bearer-gated backend authenticates via the injected token.

## Verified here
- `cd taskpaw_v3/ui && npm install && npm run build` → **green** (tsc + vite).

## Operator handoff (visual + native build)
1. Generate icons: `cargo tauri icon <a 1024px png>` (populates `src-tauri/icons/`).
2. Install the Tauri CLI: `npm i -g @tauri-apps/cli` (or `cargo install tauri-cli`).
3. Dev run: `cargo tauri dev` (from `taskpaw_v3/src-tauri/`). To have the shell
   manage the backend, set `TASKPAW_BACKEND_CMD` to the backend **executable**
   (e.g. the python interpreter or the bundled service binary) and
   `TASKPAW_BACKEND_ARGS` to its argv (e.g. `-m taskpaw_v3.agent.server.service`);
   the shell spawns it directly (no shell wrapper) so X-close terminates the real
   backend. Or run the backend separately and leave these unset.
4. Build the app: `cargo tauri build`.
5. **Visually verify** the agent console + Hub dashboard against the design system
   (dark contrast, status colors, no emoji icons, responsive), and confirm
   **X closes the window → the backend child exits** (no leftover process/port).
